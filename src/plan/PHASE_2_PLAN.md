# 街角小店第二阶段计划

## 目标

第二阶段做“经营深度扩展”：在现有单店闭环上加入顾客类型、随机事件、商品解锁、员工成长和更清晰的经营反馈。

目标不是扩大地图或增加分店，而是让玩家每天能根据“谁来买、发生了什么、该卖什么、员工怎么培养”做更有意义的选择。

## 不做的内容

- 多分店
- 后端账号
- 排行榜
- 地图探索
- 自由摆放家具
- 复杂剧情任务链
- 真实时间挂机收益

第二阶段重点是把单店经营变深，而不是扩大系统边界。

## 推荐方向

采用“顾客类型 + 随机事件 + 商品解锁 + 员工成长”的组合。

原因：

- 当前游戏已经有库存、满意度、声望、升级和自动营业。
- 现在最缺的是“为什么今天和昨天不一样”。
- 顾客类型和随机事件能直接增强每日差异。
- 商品解锁形成中期目标。
- 员工成长提供长期养成线。

## 模块 1：顾客类型系统

新增 4 类顾客：

- 学生：偏好面包、饮料，价格敏感，满意度影响明显。
- 上班族：偏好饮料、便当类新品，服务效率影响明显。
- 老街坊：偏好日用品，舒适度影响明显。
- 夜归人：偏好高价商品，库存不足惩罚更明显。

实现目标：

- 每天营业时生成顾客构成。
- 顾客类型影响商品需求、满意度变化和留言。
- 每日报告显示主要顾客群体。
- 自动营业摘要也能体现当日主要客群。

主要影响文件：

- `src/game/types.ts`
- `src/game/catalog.ts`
- `src/game/rules.ts`
- `src/game/rules.test.ts`
- `src/components/ReportPanel.tsx`
- `src/components/ShopOverview.tsx`

验收标准：

- 不同顾客类型会改变商品销量。
- 服务效率低时上班族满意度下降更明显。
- 舒适度高时老街坊贡献更高。
- 测试覆盖至少 3 种顾客影响路径。

## 模块 2：随机事件系统

新增第一批事件：

- 雨天：顾客减少，但饮料需求小幅提高。
- 放学后：学生顾客增加，面包和饮料需求提高。
- 供应商打折：当天进货成本降低。
- 街坊推荐：声望收益提高。
- 排队抱怨：服务效率不足时满意度下降。

实现目标：

- 每天营业前或营业时抽取 0-1 个事件。
- 事件影响当天需求、收入、成本、满意度或声望。
- 事件写入每日报告和顶部提示。
- 自动营业不弹窗，但摘要中显示事件名。

主要影响文件：

- `src/game/types.ts`
- `src/game/events.ts`
- `src/game/rules.ts`
- `src/game/rules.test.ts`
- `src/components/ReportPanel.tsx`

验收标准：

- 事件能被确定性测试，不依赖不可控随机。
- 每日报告展示事件名称和影响。
- 至少覆盖雨天、供应商打折、街坊推荐三个事件测试。

## 模块 3：商品解锁系统

新增商品：

- 牛奶：饮料延伸，适合学生和老街坊。
- 便当：高成本高售价，适合上班族。
- 文具：低频但稳定，适合学生。
- 鲜花：高舒适度店铺才卖得好。

解锁条件：

- 牛奶：声望 8。
- 文具：声望 14。
- 便当：店铺等级 2。
- 鲜花：装饰等级 3。

实现目标：

- 商品 catalog 支持 `unlockCondition`。
- 未解锁商品在 UI 中显示条件，不能进货。
- 解锁后进入正常进货和销售逻辑。
- 自动营业只销售已解锁商品。

主要影响文件：

- `src/game/types.ts`
- `src/game/catalog.ts`
- `src/game/rules.ts`
- `src/components/InventoryPanel.tsx`
- `src/game/rules.test.ts`

验收标准：

- 未解锁商品不能进货。
- 满足条件后商品可进货。
- 解锁商品参与每日销售。
- UI 显示明确解锁条件。

## 模块 4：员工成长系统

当前只有一名店员和固定服务加成。第二阶段改成可培养。

成长方向：

- 服务：提高服务容量。
- 补货：降低进货成本或提高容量利用。
- 亲和力：提高满意度和声望收益。

实现目标：

- 店员有经验值和等级。
- 每次营业获得经验。
- 玩家可选择一个成长方向升级。
- 员工效果进入每日结算。

主要影响文件：

- `src/game/types.ts`
- `src/game/rules.ts`
- `src/game/useGameStore.ts`
- `src/components/EmployeePanel.tsx`
- `src/game/rules.test.ts`

验收标准：

- 营业后员工获得经验。
- 经验足够时可升级一个方向。
- 服务成长提升服务容量。
- 亲和力成长提升满意度或声望。
- 测试覆盖经验增加和至少两个成长方向。

## 模块 5：经营反馈面板

新增一个轻量“今日观察”区域，避免玩家看不懂为什么赚钱变多或变少。

展示内容：

- 今日主要顾客。
- 今日事件。
- 最缺商品。
- 最大收入来源。
- 下一步建议。

主要影响文件：

- `src/components/ShopOverview.tsx`
- `src/components/InsightPanel.tsx`

验收标准：

- 玩家不打开结算弹窗也能知道自动营业发生了什么。
- 自动营业时顶部摘要和观察面板同步更新。
- 不增加新页面，不破坏移动端布局。

## 数据流

```text
catalog / events / customerTypes
  -> rules engine
    -> DailyReport
      -> useGameStore
        -> ReportPanel / ShopOverview / InsightPanel
```

## 推荐执行顺序

1. 顾客类型基础模型。
2. 顾客类型进入每日结算。
3. 随机事件模型。
4. 事件进入每日结算。
5. 商品解锁。
6. 员工成长。
7. 经营反馈面板。
8. 更新计划文档。
9. 测试和构建验证。

## 测试计划

新增测试重点：

- 指定顾客类型会改变对应商品需求。
- 服务不足时上班族满意度下降。
- 雨天事件改变需求。
- 供应商打折降低进货成本。
- 未解锁商品不能进货。
- 达成条件后商品可进货。
- 员工营业后获得经验。
- 员工成长影响服务或声望。
- 自动营业摘要包含事件和客群，但不弹窗。
- 空库存仍能自动停业。

## 风险

- 数值复杂度上升：所有倍率集中放在 `balance.ts`，不要散落在组件里。
- 随机事件难测试：规则层必须支持传入固定事件或固定随机源。
- UI 信息过载：第二阶段只加一个“今日观察”，不要把每个系统都做成大面板。
- 存档兼容：现有旧存档没有新字段，加载时需要迁移或补默认值。

## 关键决策

1. 不做多分店，先把单店做深。
2. 随机事件必须可测试，不能把 `Math.random()` 散落在规则里。
3. 商品解锁和员工成长都走现有 catalog/rules/store 架构。
4. UI 只新增经营反馈，不新增复杂页面。
5. 存档加载必须兼容旧版本。

## 完成定义

第二阶段完成时，玩家应该能明显感受到：

- 每天来的顾客不同。
- 事件会改变经营判断。
- 商品不是一次性全开放，而是随经营成长解锁。
- 店员不只是固定加成，而是可以培养。
- 自动营业时也能从界面看懂最近发生了什么。

验证要求：

```text
npm.cmd test -- --run
npm.cmd run check
```

## 执行结果

已完成：

- 顾客类型系统：学生、上班族、老街坊、夜归人进入每日结算。
- 随机事件系统：雨天、放学后、供应商打折、街坊推荐、排队抱怨进入事件 catalog，其中事件会写入报告和自动营业摘要。
- 商品解锁系统：牛奶、便当、文具、鲜花加入商品 catalog，并通过声望、店铺等级或装饰等级解锁。
- 员工成长系统：店员获得经验，可培养服务、补货、亲和力。
- 经营反馈：结算报告和店铺手账展示主要客群、今日事件和经营洞察。
- 存档兼容：旧存档加载时会补齐新增商品、员工技能和报告字段。
